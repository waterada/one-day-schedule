# release

「リリースして」と頼まれたときの一連の作業をまとめる。コミットメッセージ提案 → バンプ種別の確認 → `public/version.js` 更新 → `npx wrangler deploy`、までを順に行う。

## 手順

### 1. コミットメッセージを提案（コミットはしない）

`git status` / `git diff`（必要なら `git log` も）で直近コミットされていない変更を確認し、**日本語のコミットメッセージを 1 つ提案する**。ここではコミットを実行せず、メッセージの提案だけ行い、そのまま次のステップに進む。

- 既存コミットの文体に合わせる: 短い 1〜数行の自然な日本語、Co-Authored-By 行なし
- コミットの実行はしない。ユーザーが内容を見て自分でコミットする
- ユーザーが「これでコミットして」と続けて言ったら、そのときに実行する

この時点で変更内容がはっきりするので、次のバンプ種別判定にも使える。

### 2. バンプ種別を判定して確認

直前に確認した変更内容をもとに、以下のルールで bump 種別を提案する:

- **バグ修正のみ** → patch（`x.y.Z+1`）。例: `1.3.4 → 1.3.5`
- **機能追加・改善・UI 変更** → minor（`x.Y+1.0`、Z は 0 にリセット）。例: `1.3.4 → 1.4.0`
- 判定が曖昧なときは、変更内容を要約したうえでユーザーにどちらか確認する

`public/version.js` の現バージョンを読み取り、提案する新バージョン番号を**ユーザーに提示して、これでよいか必ず確認する**。OK が出るまで先に進まない。

### 3. `public/version.js` を更新

確認が取れたら `public/version.js` の `APP_VERSION` を新しい値に書き換える。

```js
const APP_VERSION = 'X.Y.Z';
```

シングルクォート・末尾セミコロンの形式を維持する。

### 4. デプロイ

```bash
npx wrangler deploy
```

を実行する。失敗したら出力を添えて報告し、ここで止まる。
